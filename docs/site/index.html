先做三件事

新成员第一次来，不要直接写代码。先确认身份、运行环境和认证方式，再开始接入。

01 Runtime

先确认你在哪个环境

WSL 和 Windows 原生终端的 SSH、token、path 不是一套。agent 跑在哪里，就读哪里的配置。

02 Auth

区分 SSH 和 GH_TOKEN

SSH 用于代码传输；GH_TOKEN 用于审计和 API。两者不能互相代替。

03 Workflow

所有修改都走分支和 PR

禁止把“先改 main 再说”当临时方案。治理规则的意义就是长期可追溯。

Paths

先认领你的角色，再按对应路径执行

同一个仓库，不同角色关心的问题不一样。新成员、普通开发者、管理员看到的优先级应该不同。

Contributor

普通成员共同修改项目代码

你的核心任务不是“能 push 就行”，而是让每次改动都知道基于哪个版本开始、准备合并到哪个版本。

  • 同步 tags 并从标签开分支
  • commit 必须带 Base-Version
  • push 自己的分支，不直接推 main

仓库维护者与组织管理员

管理员不仅是 merge 代码的人，更要保证项目有版本、有人接、能审计、能交接。

  • 设置 branch protection 和 required checks
  • 运行组织审计
  • 维护 OWNERS / HANDOVER
First 10 Minutes

新成员首次接入：从 GitHub 注册到治理仓库就位

这条路径专门写给“新小号 + 新 WSL 电脑”的成员。按顺序走，不要跳步。

01

注册 GitHub 并加入课题组组织

注册 GitHub 账号，绑定常用邮箱，建议开启双因素认证。然后由管理员邀请你加入 Ma-Lab2，你在邮箱或 GitHub 通知里接受邀请。

GitHub account Email verified Join Ma-Lab2
02

在新 WSL 里检查基础环境

先确认你真的在 WSL 终端里，再检查 git、python3、ssh 是否可用。如果没有，不要直接跳到 clone。

uname -a
command -v git
command -v python3
command -v ssh
03

生成 SSH key 并上传 GitHub

如果 ~/.ssh 里没有可用公钥，就生成新的。把 .pub 文件内容复制到 GitHub 的 Settings -> SSH and GPG keys。

ls -la ~/.ssh
ssh-keygen -t ed25519 -C "your-github-email@example.com"
cat ~/.ssh/id_ed25519.pub
ssh -T git@github.com
04

安装 Codex / codex cli 并确认可用

这一步只写“安装后验证”和“登录检查”。不同机器安装方式可能不同，但接入课题组前至少要确认命令存在、版本可读、账号已登录。

command -v codex
codex --version
codex login
05

用 SSH 克隆治理仓库并运行自检

治理仓库是组内协作规则入口。先把它拉下来，再用脚本确认当前 runtime、SSH、token、脚本状态。

git clone git@github.com:Ma-Lab2/laser-plasma-github-governance.git
cd laser-plasma-github-governance
./scripts/onboarding-check.sh --org Ma-Lab2 --repo laser-plasma-github-governance
06

安装 skill，再开始真正协作

治理 skill 和审计 skill 装好后，Codex 才能更稳定地按课题组规则执行。日常写代码优先用 SSH；只有跑审计、治理脚本时才需要 GH_TOKEN。

./scripts/install-governance-audit-skill.sh

# 使用时可触发
$laser-plasma-github-governance
$governance-audit
Important Split

把两类认证分清楚

很多新成员的第一个错误，就是把 SSH 和 token 混成一个概念。你必须记住：

  • SSH：日常 git clone / git pull / git push
  • GH_TOKEN：治理脚本、组织审计、GitHub API
  • 能 SSH push，不代表组织审计就一定能跑
  • 能跑 token API，不代表日常 Git 传输就该走 HTTPS
# WSL 常见做法
source ~/.bash_profile
echo ${GH_TOKEN:+SET}

# 不要把 token 发到聊天窗口
WSL recommended SSH for Git GH_TOKEN for audit

如果在 WSL 里运行，Codex 就读取 WSL 的环境变量、SSH key 和本地仓库。安装完成后，用 skill 触发词进入组内标准流程。

Claude Code / Cursor

仍然建议优先调用仓库已有脚本

无论你用的是 Claude Code 还是 Cursor，最稳妥的做法都一样：让它们读取治理仓库里的脚本和文档，不要让 agent 在空白上下文里随意发明一套流程。

  • 先跑 onboarding-check.sh
  • 先确认当前是 WSL 还是 Windows 原生
  • 先让 agent 输出执行清单，再执行
  • PR 必填 AI-Agent、AI-Assistance、Human-Review-Confirmed: yes
Workflow

课题组共同修改代码的标准流程

所有项目都要有独立版本号。多人维护时，最怕的不是写得慢，而是基线不清楚、合并后没人知道代码基于哪个版本继续演化。

普通成员 6 步法

  1. 同步 tags
  2. 从明确 tag 开分支
  3. 开发和测试
  4. commit 时写 Base-Version
  5. push 自己的分支
  6. 提 PR，等待 review 和合并
git fetch --tags --prune
./.governance/start-from-tag.sh v0.1.0 feature/your-change

git add .
git commit -m "your change summary" -m "Base-Version: v0.1.0"
git push origin feature/your-change

PR 里必须补齐的信息

  • Base-Version
  • Change-Type
  • Target-Version
  • AI-Agent
  • AI-Assistance
  • Human-Review-Confirmed: yes
  • AI-Notes

管理员需要额外保证：branch protection 开启、required checks 就位、OWNERS 与 HANDOVER 可用、组织审计可跑。

Admin

管理员的职责不是“能 merge”，而是“让项目可持续”

如果一个项目只靠一个学生维护，等他毕业后就会断代。管理员要负责把仓库变成可交接系统，而不是个人文件夹。

Repository Baseline

新仓库初始化

给新项目安装治理模板、替换占位符、刷新 lock 信息、建立初始标签，再配置最小 branch protection。

./scripts/apply-governance-template.sh /path/to/target-repo 0.2.1
python3 .governance/update-skill-lock.py --skill-version 0.2.1
git tag v0.1.0
git push origin v0.1.0
Organization Audit

定期检查所有仓库是否按治理规则运行

组织审计的意义不是“挑错”，而是尽早发现没有版本号、没有治理模板、没有 required checks、没人接手的仓库。

./skills/governance-audit/scripts/run-org-audit.sh \
  --org Ma-Lab2 \
  --fail-on-blocker false
