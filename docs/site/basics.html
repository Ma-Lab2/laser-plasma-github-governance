GitHub Basics

先把 GitHub 基础概念讲清楚，再谈协作规范

如果成员只知道“把代码传上去”，却不知道 repo、branch、commit、tag、PR 分别是什么，后面的治理规则就很容易变成机械背诵。这里先把最基础的对象和它们之间的关系讲清楚。

看协作流程如何落地 先去做首次接入
One Sentence

GitHub 不是网盘，是可追踪的协作历史系统。

它记录谁在什么时候基于哪个版本做了什么改动，并通过分支、PR 和 review 把多人工作组织起来。

Core Objects

最核心的 8 个概念

Repository

一个项目的代码仓库。包含代码、文档、提交历史、分支、标签、PR、Issue 等。

Clone

把远端仓库复制到本地电脑。之后你在本地写代码，再与远端同步。

Branch

并行开发线。你应该在自己的功能分支上工作，而不是直接改 main。

Commit

一次有边界的修改记录。好的 commit 让人能回看“这个改动到底干了什么”。

Tag

给某个稳定提交打上版本标签，例如 v0.2.1。在本课题组里，开发必须从明确 tag 开始。

Push / Pull

push 是把本地分支传到远端；pull 是把远端最新变化拉回本地。

Pull Request

把一个分支的变更提交给目标分支审核。PR 是 review、讨论、CI 校验和合并的中心。

Review

别人检查你的改动是否正确、是否符合规范。对科研代码来说，review 是长期可维护性的核心。

Mental Model

用一条演化链理解 GitHub

01

仓库从稳定版本开始

仓库有一个稳定版本，比如 v0.1.0。这不是装饰，而是协作基线。

02

每个人从基线拉出自己的分支

这样不同成员的工作不会直接互相覆盖，彼此可以独立推进。

03

分支上的改动通过 commit 形成历史

commit 把“正在做的事情”切成可解释、可回滚、可审查的单元。

04

通过 PR 进入集体审核

PR 是把个人修改纳入团队标准流程的入口。

05

合并后形成新的稳定历史

如果需要发版本，再更新 VERSION、CHANGELOG 并打新的 tag。

Common Confusions

最常见的概念混淆

Branch 不是仓库副本

分支仍属于同一个仓库，只是同一历史上的不同开发线。你不需要为每个小功能新建一个仓库。

Commit 不是“保存一下”

commit 应该代表一个可说明的改动单元。尤其在课题组治理里，commit 还要带版本基线信息。

Tag 不是 branch

tag 是某个历史点的稳定标记，不是继续开发的活动线。所以正确做法是从 tag 开 branch。

PR 不是可有可无的形式

PR 里集中了 review、讨论、CI 结果和 AI 使用说明。没有 PR，就失去了协作和审计上下文。
