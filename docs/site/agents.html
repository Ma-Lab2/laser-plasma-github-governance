Agent Rules

AI agent 可以提高效率，但不能绕过治理规则

不管你用的是 Codex、codex cli、Claude Code 还是 Cursor，本质都一样：agent 运行在哪个环境，就继承哪个环境的 SSH、token 和仓库状态；最终改动仍然要走 branch、PR、review 和版本规则。

Remember

先判断 agent 跑在 WSL 还是 Windows

这是所有认证问题的第一分水岭。不要在 Windows 终端里假设它能读取 WSL 的 GH_TOKEN 或 ~/.ssh。

Codex

Codex / codex cli 接入

先确认命令和登录状态

command -v codex
codex --version
codex login

如果这些都过不了，就不要先让 Codex 操作仓库。

再接治理 skill

./scripts/install-governance-audit-skill.sh

# 在 Codex 中可触发
$laser-plasma-github-governance
$governance-audit

让 agent 优先读取现有脚本和文档，不要凭空造新的治理流程。

Cross-Tool Rules

所有 agent 的共通规则

先跑环境检查

先用 ./scripts/onboarding-check.sh 确认 runtime、SSH、token 和仓库状态。

先给 agent 明确目标

例如让它“先检查 SSH 是否可用，再克隆治理仓库，再运行自检”，不要一句“帮我全部搞定”没有边界。

先输出执行清单再执行

对于新机器、生产仓库、组织治理这种高影响操作，先让 agent 列步骤，能减少误改配置的风险。

PR Policy

AI 改动也必须披露

  • AI-Agent
  • AI-Assistance
  • Human-Review-Confirmed: yes
  • AI-Notes

为什么需要人工确认

因为 AI 可以加速实现，但不能替代对科学代码、版本兼容性和协作影响的最终判断。合并责任仍然在人类维护者身上。
