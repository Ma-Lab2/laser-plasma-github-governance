Contributor Workflow

在课题组仓库中共同修改代码的标准方法

普通成员最重要的不是会多少 Git 命令，而是按正确顺序做正确的事：先同步 tags，再从标签开分支，commit 时带版本基线，push 自己的分支，最后通过 PR 合并。

Core Rule

不要直接改 main

如果你有写权限，也不代表你应该直接推主分支。治理规则的设计就是为了避免多人同时破坏主线历史。

6 Steps

普通成员 6 步法

01

同步 tags

先拿到所有版本标签，否则你根本不知道自己基于哪个稳定版本开始工作。

git fetch --tags --prune
02

从明确 tag 开分支

你必须知道自己是从 vX.Y.Z 的哪个稳定基线开始的，而不是直接基于某个飘忽不定的最新提交。

./.governance/start-from-tag.sh vX.Y.Z feature/your-branch
03

开发并本地测试

在自己的分支上写代码、改文档、修 bug。不要把尚未整理的实验性修改直接推上去。

04

commit 时写 Base-Version

每个 commit 都要明确写出它基于哪个版本开始，这样后续 review 和 release 才有依据。

git add .
git commit -m "your change summary" -m "Base-Version: vX.Y.Z"
05

push 自己的分支

推送的是你的功能分支，不是默认分支。

git push origin feature/your-branch
06

提 PR 并等待 review

PR 里必须补齐治理字段，等 review 和 CI 全部通过后再合并。

PR Fields

PR 必填内容

  • Base-Version
  • Change-Type
  • Target-Version
  • AI-Agent
  • AI-Assistance
  • Human-Review-Confirmed: yes
  • AI-Notes

为什么要填这些

因为多人维护时，最怕的是“谁也不知道这次改动基于哪个版本开始”“AI 改了多少没人说得清”“合并后版本应该往哪走没有共识”。这些字段就是为了解决这些问题。

Common Failures

最容易被拒的情况

缺 Base-Version

重新整理 commit message，补 trailer 后再 push。

改了 VERSION 却没改 CHANGELOG

说明版本演化记录不完整，必须补齐。

直接推默认分支

这不是捷径，是破坏治理规则。正确做法永远是 branch + PR。
